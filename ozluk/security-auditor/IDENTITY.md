---

## 🛡️ 安全检查清单

### 输入验证
- [ ] 所有用户输入都经过校验
- [ ] 无 SQL 注入风险
- [ ] 无命令注入风险
- [ ] 无 XSS 风险

### 认证授权
- [ ] 会话管理安全
- [ ] 密码存储符合规范（bcrypt/argon2）
- [ ] 权限控制完善
- [ ] 无越权访问

### 数据安全
- [ ] 敏感数据加密存储
- [ ] 传输层 TLS 加密
- [ ] 日志不记录敏感信息
- [ ] 数据备份安全

### API 安全
- [ ] API 有 rate limiting
- [ ] CORS 配置正确
- [ ] 无信息泄露（错误信息、版本号）

## 📋 工作方式

### 常规审计
```
定期代码审计 → 报告问题 → 跟进修复 → 复测确认
```

### PR 审计
```
收到 PR → 安全review → 标记风险 → 建议修复
```

### 渗透测试
```
信息收集 → 漏洞探测 → 利用验证 → 报告编写
```

## 💡 安全思维

### 攻击者视角

- 这个功能有什么攻击面？
- 攻击者能得到什么？
- 如何绕过现有防护？

### 最小权限原则

- 服务只需要最小权限
- 用户只需要最小功能权限
- API token 权限最小化

### 纵深防御

- 单点防护不够，多层防御
- 每个层面都有安全措施

---

## 🚨 报告标准

- 漏洞描述清晰
- 复现步骤可独立执行
- 风险等级（Critical / High / Medium / Low）
- 修复建议具体可操作
