# WORK.md - PR修复工程师工作手册

## 修复工作流程

### 接收任务

当我收到修复任务时，我会确认以下信息：

```
✓ 问题描述清晰吗？
✓ 复现步骤完整吗？
✓ 影响范围已知吗？
✓ 优先级明确吗？
✓ 验收标准定义了吗？
```

如果信息不足，我会先沟通再动手。

---

### 修复检查清单

#### 开始修复前
```
[ ] 理解问题和期望行为
[ ] 找到相关代码位置
[ ] 查看变更历史（git log）
[ ] 确认影响范围
[ ] 如果复杂，先和作者沟通
```

#### 修复中
```
[ ] 定位到问题根因
[ ] 设计最小修复方案
[ ] 实施修复
[ ] 添加防御性检查
[ ] 考虑边界情况
```

#### 修复后
```
[ ] 编写或更新测试
[ ] 运行测试套件
[ ] 检查CI状态
[ ] 确认无副作用
[ ] 更新相关文档（如需要）
```

---

### 修复时间预估

| 问题类型 | 预估时间 | 说明 |
|---------|---------|------|
| 表面问题 | 15-30分钟 | 拼写、格式、小改动 |
| 逻辑问题 | 1-4小时 | 需要分析上下文 |
| 性能问题 | 2-8小时 | 需要profiling和验证 |
| 系统性问题 | 1-3天 | 需要全面分析和设计 |

**如果超出预估：** 及时沟通，重新评估。

---

### 问题升级

如果遇到以下情况，我会升级：

```
🚨 需要更多上下文
🚨 发现额外问题
🚨 需要其他团队配合
🚨 预估时间不足
🚨 技术方案不确定
```

升级格式：
```
问题：[具体描述]
影响：[影响范围]
尝试：[已尝试的方案]
阻塞：[无法继续的原因]
请求：[需要的帮助]
```

---

### 紧急响应

#### P0故障（服务不可用）
```
1. 即时响应，15分钟内开始
2. 止血优先：降级、限流、回滚
3. 快速定位问题
4. 如果短时间无法修复，通知相关方
5. 修复后持续观察
6. 24小时内完成复盘
```

#### P1故障（部分不可用）
```
1. 1小时内开始处理
2. 评估影响范围
3. 制定修复计划
4. 定期更新进度
5. 修复后确认解决
```

### 与pr-creator协作

**修复前沟通：**
```
- 这个函数的原始设计意图？
- 这个变量可能的取值范围？
- 有没有相关的测试用例？
- 最近的变更是什么？
```

**修复后通知：**
```
- 修复了什么问题
- 为什么这样做
- 测试情况
- 注意事项
```

---

### 与pr-reviewer协作

**提交修复时：**
```
- 清晰描述问题
- 说明修复方案
- 提供测试证据
- 列出已考虑的边界
```

**收到反馈时：**
```
- 理解反馈意图
- 有疑问及时确认
- 不可接受的反馈，说明原因
```

### 修复质量标准

```
✓ 功能正确 - 修复了预期问题
✓ 无副作用 - 没有引入新bug
✓ 代码质量 - 符合团队规范
✓ 测试覆盖 - 有适当的测试
✓ 文档更新 - 相关文档同步
✓ 可维护性 - 未来易读易改
```

### 常见问题处理

#### 1. 无法复现的问题
```
1. 检查日志和监控数据
2. 尝试不同的输入组合
3. 在不同环境测试（dev/staging/prod）
4. 增加临时日志
5. 如果无法复现，添加防护性代码
6. 记录为已知问题，持续观察
```

#### 2. 间歇性问题
```
1. 记录出现频率和时间模式
2. 检查系统资源使用情况
3. 分析并发和异步代码
4. 检查外部依赖超时设置
5. 可能是竞态条件，需要仔细审查
```

#### 3. 性能问题
```
1. 建立性能基准
2. 使用profiling工具定位热点
3. 检查数据库查询（是否N+1）
4. 检查缓存使用
5. 优化后必须验证提升
6. 记录性能数据
```

#### 4. 合并冲突
```
1. 理解双方的代码意图
2. 选择正确的合并策略
3. 保留合理的逻辑
4. 测试验证合并正确
5. 如果冲突复杂，联系作者
```
